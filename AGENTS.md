# Repository Guidelines

## Project Structure & Module Organization
- `src/` contains the TypeScript source (core URLPattern generation logic).
- `tests/` holds the two test suites (`manual.test.ts`, `generated.test.ts`).
- `tests/fixtures/` contains fixture data, transforms, and setup files.
- `index.html` is the manual browser test tool.
- `dist/` is build output (published artifacts).
- `coverage/` is generated by test runs.

## Build, Test, and Development Commands
- `npm run dist` builds the library with Vite (outputs to `dist/`).
- `npm start` serves the workspace on `http://localhost:8000` (useful for `index.html`).
- `npm test` runs Vitest in both Node and browser projects.
- `npm run test:node` runs Node-only tests.
- `npm run test:browser` runs Chromium browser tests.
- `npm run test:types` runs TypeScript typechecking.
- `npm run lint` checks code style with ESLint.
- `npm run lint:fix` auto-fixes lint issues where possible.
- `npm run transform:testdata` regenerates `tests/fixtures/fixture-data.ts` from the downloaded URLPattern dataset in `tests/fixtures/raw-fixtures/`.

## Coding Style & Naming Conventions
- TypeScript is written in ES module style with `type` exports where needed.
- Formatting is handled by Prettier (`.prettierrc`) and linted with ESLint.
- Naming: `camelCase` for variables/functions, `PascalCase` for types/interfaces.
- Keep new UI tweaks in `index.html` consistent with existing CSS variables.

## Testing Guidelines
- Framework: Vitest (projects configured in `vitest.config.ts`).
- Tests live in `tests/*.test.ts` and follow `*.test.ts` naming.
- Coverage thresholds are 100% for statements/branches/functions/lines.
- Fixture tests come from a downloaded URLPattern dataset; `transform:testdata` builds the generate fixtures, so `tests/fixtures/fixture-data.ts` should not be edited by hand.
- Run `npm test` before submitting changes.

## Commit & Pull Request Guidelines
- Commit messages are short, imperative summaries (e.g. “Cleanup dev files”, “Note inconsistencies across browsers and node”).
- PRs should include:
  - A brief summary of changes.
  - Any behavior changes or edge cases.
  - Screenshots for `index.html` UI updates.

## Manual Testing
- Open `index.html` in a browser for the explorer UI.
- Use the pattern tabs and exec section to validate URLPattern behavior against `generate`.
